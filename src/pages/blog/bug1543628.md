## Intro

This was a particularly old [bug](https://bugzilla.mozilla.org/show_bug.cgi?id=1543628), dating from more than 6 years ago. Throughout the years a few developers took a stab at fixing it, but - as we will see - the [bug](https://bugzilla.mozilla.org/show_bug.cgi?id=1543628) had a tendency of creeping out and necessitating evermore changes, which ended up stalling the efforts. Meanwhile Florens Verschelde (:fvsch), the maintainer, left valuables clues in his comments on the [Bugzilla's page](https://bugzilla.mozilla.org/show_bug.cgi?id=1543628).

To understand the problem it's worth reminding that the devtools panels - Console, Debugger, Netmonitor, and others - also need CSS styling to be rendered. Everything you see on your Firefox browser - including the [chrome]() has to be stylized at some point, even though we, as users, rarely think about it. (this might be a good segway for Mike Conley's video).

The Debugger panel used a React component called `AccessibleImage` to render icons (only the Debugger panel or was it shared?). Here's how it worked:
```js
  const AccessibleImage = props => {
    return React.createElement("span", {
      ...props,
      className: classnames("img", props.className),
    });
  };
```

The classnames utility combines CSS class names into a single string. So when you wrote:
```html
<AccessibleImage className="close" />

```

React rendered:
```html
<span class="img close"></span>

```

The component always added `img` as a base class, then appended whatever you passed via `className`. The corresponding CSS used these classes as selectors:
```css
.img {
width: 16px;
height: 16px;
background-color: var(--theme-icon-color);
}

.img.close {
mask-image: url(debugger/images/close.svg);
}

```

The selector `.img.close` targets elements that have both classes. However, CSS class selectors are global. When the browser sees `.img`, it matches any element with that class - regardless of which panel's code created it. Class names like img, close, worker, file, and arrow are generic enough that other panels might reasonably use them too. If Console styles `.close` for its own purposes, those styles could bleed into Debugger icons when components are shared.

The solution is **namespacing** - prefixing class names to make them unique to their context. Instead of `.img` and `.close`, we use `.dbg-img` and `.dbg-img-close`. This convention makes conflicts virtually impossible while keeping the intent clear.

## The Problem

Images in Debugger (devtools panel) were being styled like this:

```css
.img {
  width: 16px;
  height: 16px;
  background-color: var(--theme-icon-color);
}

.img.foo {
  mask: url(debugger/images/foo.svg);
}

.img.bar {
  mask: url(debugger/images/bar.svg);
}
```
A class name like `img` is far too generic to style elements from a panel from devtools - namely the Debugger. This makes clashes with other devtools styling classes too likely to occur. There's no reason to think devtools panels like Console or Netmonitor wouldn't need an `img` class for their CSS styling as well.

Tackling this problem would involve 3 steps:

1. Renaming the AccessibleImage component to just Image or DebuggerImage (it doesn't have any accessibility feature; we could add the ability to pass alt text, though.).
2. Changing the base class from img to something a bit less generic, for instance dbg-img.
3. Changing all the image specific class names from <name> to dbg-img-<name>, for instance: dbg-img-worker.

## Fixing Step-By-Step

### 1. Modifying the component

Rename the AccessibleImage component to just Image or DebuggerImage (it doesn't have any accessibility feature; we could add the ability to pass alt text, though.)

### 2. Changing the base class

Change the base class from img to something a bit less generic, for instance dbg-img.

### 3. Changing class names

Change all the image specific class names from <name> to dbg-img-<name>, for instance: dbg-img-worker.
