## Intro

This was a particularly old [bug](https://bugzilla.mozilla.org/show_bug.cgi?id=1543628), dating from more than 6 years ago. Throughout the years a few developers took a stab at fixing it, but - as we will see - the [bug](https://bugzilla.mozilla.org/show_bug.cgi?id=1543628) had a tendency of creeping out and necessitating evermore changes, which ended up stalling the efforts. Meanwhile Florens Verschelde (:fvsch), the maintainer, left valuables clues in his comments on the [Bugzilla's page](https://bugzilla.mozilla.org/show_bug.cgi?id=1543628).

To understand the problem it's worth reminding that the devtools panels - Console, Debugger, Netmonitor, and others - also need CSS styling to be rendered. Everything you see on your Firefox browser - including the [chrome]() has to be stylized at some point even though we, as users, rarely think about it. (this might be a good segway for Mike Conley's video).

One of the devtools panels, the Debugger, used a React component called `AccessibleImage` to render icons. Here's how it was defined:
```js
  const AccessibleImage = props => {
    return React.createElement("span", {
      ...props,
      className: classnames("img", props.className),
    });
  };
```

The `classnames` utility combines CSS class names into a single string. So when you wrote:
```html
<AccessibleImage className="close" />

```

React rendered:
```html
<span class="img close"></span>

```

The component always added `img` as a base class, then appended whatever you passed via `className`. So the corresponding CSS would use these classes as selectors:
```css
.img {
width: 16px;
height: 16px;
background-color: var(--theme-icon-color);
}

.img.close {
mask-image: url(debugger/images/close.svg);
}

```

The selector `.img.close` targets elements that have both classes.

## The Problem

The choice of `.img` for the base class name of icons generated by `AccessibleImage` is a problematic one. As you can probably feel, it sounds too generic... Think of all other panels that will be sharing this **namespace** (Console, Debugger, Netmonitor, etc). If a developer forgets about how `AccessibleImage` icons are rendered in the Debugger panel and accidentaly creates a `.img` class for CSS selector usage in another panel, there'll be a conflict.

(background: [Debugger CSS uses global classNames, risks conflicts](https://github.com/firefox-devtools/debugger/issues/7905))
(the below example sounds too trivial and hopefully the link above can help provide better justification for the problem)

For example, if images in Debugger are styled like this:

```css
.img {
  width: 16px;
  height: 16px;
  background-color: var(--theme-icon-color);
}

.img.foo {
  mask: url(debugger/images/foo.svg);
}

.img.bar {
  mask: url(debugger/images/bar.svg);
}
```
Then Console, needing to style some images of its own, might write:
```css
.img {
  border: 2px solid red;
  width: 32px;
  height: 32px;
}
```

And now any Debugger icons pick up Console's styles - wrong size, unexpected border... The Debugger styles and Console styles are both valid on their own, but they're fighting over the same class name.

There's also a different kind of clash that these overly generic CSS selectors facilitate. To understand it, let's look into a concrete example. Say the Threads pane displayed a list of web workers with this structure:
```html
  <div class="workers-list">
    <div class="worker">
      <div class="icon">
        <span class="img worker"></span>
      </div>
      <div class="label">...</div>
    </div>
  </div>
```

Notice that `worker` appears twice - once on the list item (`div.worker`) and once on the icon (`span.img.worker`). The CSS for highlighting a row on hover looked like this:
```css
  .workers-list .worker:hover {
    background-color: var(--search-overlays-semitransparent);
  }
```

The selector `.workers-list .worker:hover` matches any `.worker` descendant - including the icon. On hover, both the row and the icon received the background color change. This made the icon (`<span class="img worker"></span>`) shift to an unintended color.

## The Approach

To deal with this we use [namespacing](https://developer.mozilla.org/en-US/docs/Glossary/Namespace). We prefix class names to make them unique to their context. Instead of `.img` and `.close`, we use `.dbg-img` and `.dbg-img-close`. This convention makes conflicts virtually impossible while keeping the intent clear.

In the devtools Debugger panel context, [namespacing](https://developer.mozilla.org/en-US/docs/Glossary/Namespace) our image class involves taking these 3 steps: (any more reasoning to devise this engineering strategy?)

1. Renaming the `AccessibleImage` component to `DebuggerImage` and providing the latter with a proper API to achieve the CSS [namespacing](https://developer.mozilla.org/en-US/docs/Glossary/Namespace) we're after.
2. Changing the base class from `img `to something a bit less generic, for instance `dbg-img`.
3. Changing all the image specific class names from `<name>` to `dbg-img-<name>`, for instance: `dbg-img-worker`.

## Step-By-Step Implementation

### 1. Modifying the component

Rename the AccessibleImage component to just Image or DebuggerImage (it doesn't have any accessibility feature; we could add the ability to pass alt text, though.)

### 2. Changing the base class

Change the base class from img to something a bit less generic, for instance dbg-img.

### 3. Changing class names

Change all the image specific class names from <name> to dbg-img-<name>, for instance: dbg-img-worker.
