## Intro

As this bug had many previous attempts at solving it, its respective [Bugzilla page](https://bugzilla.mozilla.org/show_bug.cgi?id=1730474) was jam-packed. It's always helpful to access information pertaining to the bug you're trying to fix, but in large quantities it can take some time to digest. Fortunately, Eitan Isaacson (:eeejay), the bug's mentor, did a great job guiding the efforts for the solution.

To understand this bug, we got to delve a bit into the concept of [accessibility tree](https://developer.mozilla.org/en-US/docs/Glossary/Accessibility_tree). The [accessibility tree](https://developer.mozilla.org/en-US/docs/Glossary/Accessibility_tree) is a parallel representation of a web page that's specifically designed for assistive technologies (like screen readers, voice control software, etc.). Like a simplified, semantic version of the DOM. Without the [accessibility tree](https://developer.mozilla.org/en-US/docs/Glossary/Accessibility_tree), screen readers would have to parse raw HTML/DOM, which contains too much noise. The tree provides a clean, semantic interface that assistive technologies can reliably consume.

As an example, let's say you have this HTML:
```
  <div class="card-wrapper">
    <div class="card">
      <button id="btn">Click me</button>
    </div>
  </div>
```

Its DOM tree will be:
```
  div (class="card-wrapper")
    └─ div (class="card")
        └─ button#btn
```

And its (somewhat simplified) Accessibility tree would look like:
```
  button
    ├─ role: button
    ├─ name: "Click me"
    ├─ state: focusable
    └─ state: enabled
```

Each node in the [accessibility tree](https://developer.mozilla.org/en-US/docs/Glossary/Accessibility_tree) has:

  - Role - What type of element is this? (button, heading, table, cell, navigation, etc.)
  - Name - What's it called? (button text, alt text, aria-label, etc.)
  - State - Current condition (checked, expanded, selected, focused, etc.)
  - Properties - Static attributes (readonly, required, level, etc.)
  - Relationships - Connections to other elements (labelledby, describedby, owns, etc.)

The default role of a native `<td>` element in a native HTML table (without ARIA role overrides) is `cell`. Notably, this role is absent from the "Used in Roles" values in the [ARIA specification](https://www.w3.org/TR/wai-aria-1.1/#aria-selected) page. This means that by default, the `aria-selected` state value for `cell` nodes in the a11y tree should be `undefined`. Having it `false` or `true` would imply they're selectable.

## The Problem

The bug was that Firefox was making all table cells expose the SELECTABLE state in the [accessibility tree](https://developer.mozilla.org/en-US/docs/Glossary/Accessibility_tree). This made screen readers announce every single table cell as "selectable" when navigating regular HTML tables (like Wikipedia tables). This problem emanates from code that adds `states::SELECTABLE` flag irrespective of any circumstance.

Solving this problem would involve 3 steps:
1. Going to its root and making native HTML table cells NOT expose SELECTABLE state by default.
2. Creating a test in the a11y test suite that confirms this behavior have been fixed.
3. Checking if this broke any previous test and, if so, understanding why it did and fixing it.

## A 3-Layers Solution

### 1. Removing the override

The culprit code was located in the `accessible/html/HTMLTableAccessible.cpp` file:

```
uint64_t HTMLTableCellAccessible::NativeInteractiveState() const {
  return HyperTextAccessible::NativeInteractiveState() | states::SELECTABLE;
}
```
This code was basically ensuring that all HTML table cells were selectable. (explanation of the mechanism through which C++ state, bits, and inheritance control the aria states of HTML cells)

The `NativeInteractiveState` override function...

### 2. Creating the test file

To prove that the bug was fixed, I needed to create a test covering this particular issue. Following all conventions set in the codebase, I came up with the following JavaScript test:
```
/**
 * Test that native HTML table cells are not selectable by default.
 */
addAccessibleTask(
  `
  <table id="native_table">
    <tr>
      <td id="native_cell">Native cell</td>
    </tr>
  </table>
  `,
  async function (browser, docAcc) {
    const nativeCell = findAccessibleChildByID(docAcc, "native_cell");
    testStates(nativeCell, 0, 0, STATE_SELECTABLE, 0);
  },
  { topLevel: true, iframe: true, remoteIframe: true, chrome: true }
);
```
It's very simple and self-explanatory; it simply tests that a native HTML table cell is not going to have that `SELECTABLE` state set as `true` by default.

### 3. Fixing test regressions

After fixing this problem and devising a test to prove it, I expected some previously passing tests to fail. That's because the tests were built around this erronous behavior, i.e. of all native HTML table cells being selectable by default. It was paramount to run the test suite and see what tests would fail in development. I needed not only to confirm my code changes, but to find any test that broke due to those changes. Fortunately I was able to [overcome my hardware limitations](/blog/building-mozilla-on-a-droplet) and run the testing suite.

The assertions from file `accessible/tests/mochitest/table/test_sels_table.html` were then failing because it was expected that their cells selectable state would default to `false`. With this patch they should now default to `undefined`. This change, moreover, complies with [ARIA specifications](link to the said ARIA spec).

Luckily this was a pretty self-contained regression issue, and simply changing the values `false` to `undefined` on a couple arrays (assigned to `cellArray` variables) in this single test file sufficed.
